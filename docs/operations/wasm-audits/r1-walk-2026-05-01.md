# r1 WASM walk — comprehensive history

> Snapshot of the actual WASM hash timeline for every Soroban
> source we ingest, captured by walking r1's full Galexie archive
> (`s3://galexie-archive`, ledgers 50,457,424 → 62,296,694) on
> 2026-04-30 and re-analysed 2026-05-01. Supersedes the
> per-source `wasm-audits/*.md` files where they diverge from the
> walk; those files retain audit-trail context but their hash
> citations may be stale.

## What this answers

For every protocol/oracle/etc. that produces canonical trades:

1. **Which contracts are deployed on mainnet** (factory, router, per-instance pools, oracles).
2. **When each was deployed** (first-seen ledger).
3. **When each was upgraded** (every `update_current_contract_wasm` event, rendered as a contiguous ledger range per WASM hash).
4. **What each upgrade resulted in** (factory upgrades cascade — pool instances deployed *after* a factory upgrade adopt the new template; existing instances stay on their old WASM until they're individually upgraded).

## How the walk was performed

Run on r1 (`ssh root@136.243.90.96`) on 2026-04-30:

```sh
ratesengine-ops wasm-history \
  -config /etc/ratesengine.toml \
  -from 50457424 -to 62296694 \
  -contracts <532-contract-list> \
  -parallel 8 \
  > /var/log/wasm-history-full.json
```

The 532-input list was assembled from the per-source contract
registries plus stellar.expert `deploy` event enumeration for
factories. Walk took ~10 hours wall-clock; per-worker JSONL
checkpoints in `/var/log/wasm-history-checkpoints/`.

A second curated run produced
`/var/log/wasm-history-all.json` (13 anchor contracts only) for
quick reference.

The full output JSON, the per-source breakdown, and the analysis
script live in
[`evidence/r1-walk-2026-05-01/`](evidence/r1-walk-2026-05-01/).

The 283 "no transitions" contracts include those deployed before
the walk's `from-ledger` (50,457,424) without any subsequent
upgrade — the wasm-history walker only records *transitions*,
so a contract that's been on the same WASM since before walk
start emits no ranges. They're not absent from mainnet; they're
just stable.

## Tooling gaps surfaced

1. **`ratesengine-ops extract-wasm-from-galexie` returns
   `MISSING` for every hash.** Tested 2026-05-01 against a
   subset of the 47 hashes; tool reports zero matches across
   the entire archive. Likely root cause: the `Created`/`Restored`
   `LedgerEntryChange` matching in `cmd/ratesengine-ops/wasm_extract.go:284`
   isn't catching `UploadContractWasm` host-fn invocations
   (perhaps the change comes through as `Updated` or a different
   discriminator under the V3/V4 metadata variants the
   `wasm-history` walker handles fine). Until this is fixed, we
   cannot preserve raw WASM bytes on-disk under
   `evidence/<source>/`. The hash inventory above is still
   trustworthy because the `wasm-history` tool — which uses a
   *different* code path against the same archive — reports
   coherent timelines. **Open as a separate issue.**

2. **The wasm-history walk's input list isn't curated in repo.**
   The 532-contract input was assembled ad-hoc on r1; nothing
   in the repo describes which contract IDs went in or where they
   came from. A future re-run would have to reconstruct the list
   manually. Cure: ship a curated inventory file at
   `configs/audit/wasm-walk-contracts.yaml` per source, with
   per-source enumeration commands documented.

3. **`ranges: null` semantics are ambiguous.** A null result can
   mean (a) contract genuinely never changed during the walk, or
   (b) contract was deployed pre-walk-window without subsequent
   upgrade — these are operationally identical but
   epistemologically different. The walker should emit at least
   the *current* hash for null contracts so we can distinguish.
