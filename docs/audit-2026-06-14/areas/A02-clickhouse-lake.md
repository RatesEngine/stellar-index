## CORRECT — verified-good (explicitly checked, no finding)

**Fresh code — `extract_entry_changes.go`:**
- **Mirrors `dispatcher.walkEntryChanges` exactly.** FeeChanges → `op_index -1`; `TxChangesBefore` → -1; per-op `v3.Operations[i].Changes` / `v4.Operations[i].Changes` → their `opIdx`; `TxChangesAfter` → -1. V3 via `MustV3()`, V4 via `MustV4()`; V1/V2 (pre-Soroban) fall through the `switch` with no `default` → no rows, matching the dispatcher's skip. `OperationMetaV2.Changes` confirmed present in SDK v0.5.0 (`xdr_generated.go:18455`), so `v4.Operations[opIdx].Changes` is the right field.
- **`op_index -1` correctly typed.** `emit` casts to `int32(opIndex)`; the row field `OpIndex int32` (sink.go:98) and the schema column `op_index Int32` (tier1_schema.sql:133) both hold -1. Tested (`extract_entry_changes_test.go` asserts -1 round-trips).
- **Key/entry XDR marshal.** created/updated/state → `entry.LedgerKey()` for the key + `MarshalBase64(entry)` for the entry body; removed → `c.GetRemoved()` key only, no entry XDR (tested). Each marshal error returns `ok=false` → the change is skipped + tolerated, never fatal (matches `ExtractLedger`'s resilience contract). `default` arm returns `ok=false`.
- **change/entry type name maps** are total over the XDR enums (both have an `"unknown"` default).

**Fresh code — `explorer_reader.go`:**
- **FINAL usage is correct per the stated rule.** FINAL is used ONLY on ledger-keyed reads that don't rely on a bloom index (`RecentLedgers`, `LedgerBySeq`, `LedgerTransactions`, `OperationsByLedger`, `OperationsByTx`, `EventsByTx`, `OperationResultsByTx` — all partition-pruned by `ledger_seq` and/or PK-prefix). FINAL is correctly OMITTED on the three bloom-index lookups (`TransactionByHash`/tx_hash, `ContractEventsRecent`/contract_id, `AccountTransactions`+`AccountOperations`/source_account) so the skip-index isn't defeated — with docstrings explaining why. (The residual duplicate-row caveat for the list endpoints is the Medium finding above; FINAL-vs-bloom itself is right.)
- **All user-influenced values go through `?` binds**, never `fmt.Sprintf` with request data: `beforeSeq`, `seq`, `hash`, `account`, `contractID`, `limit`, `beforeLedger` are all positional `?` args. No SQL injection on the explorer read path.
- **Input validation upstream:** account → `canonical.IsAccountID` (G-strkey); contract → `canonical.IsContractID` (C-strkey); tx hash → `^[0-9a-f]{64}$` regex after lowercasing. Invalid input → 400 before the reader is touched.
- **Limit caps everywhere.** Every list method clamps `limit` to a sane default + hard max (ledgers 200, txs 1000, ops 2000, account 200, contract events 500), and the API layer additionally validates `?limit=` to `[1,maxN]`. No unbounded result sets.
- **Pool sizing** is request-appropriate (MaxOpen 8 / MaxIdle 4, 30s read timeout, 30s `max_execution_time`) — distinct from the streaming `openRead` profile. Ping-on-construct.
- **Scan-target matching:** `scanLedger`/`scanTxSummaries`/`scanOps` column lists match the `SELECT` column constants 1:1; `successful UInt8` scanned into a `uint8` then `!= 0` → bool (the schema is UInt8, not Bool — correct). `total_coins`/`fee_pool` scanned as `int64` then serialised as decimal STRINGS at the wire (`explorer_ledgers.go:40-41`, `strconv.FormatInt`) → ADR-0003 satisfied (stroops fit int64; JSON never sees a >2^53 number).
- **nil/error handling:** every `Query` checks err; `rows.Close()` deferred; loops return `rows.Err()`; not-found is `(zero, false, nil)` not an error.

**Schema — `tier1_schema.sql`:**
- **ReplacingMergeTree dedup keys = each row's natural identity:** ledgers `ORDER BY ledger_seq`; transactions `(ledger_seq, tx_index)`; operations `(ledger_seq, tx_index, op_index)`; operation_results `(ledger_seq, tx_hash, op_index)`; contract_events `(ledger_seq, tx_hash, op_index, event_index)`; ledger_entry_changes `(ledger_seq, tx_hash, op_index, change_index)`; supply_flows `(contract_id, ledger_seq, tx_hash, op_index, event_index)` (contract_id first → fast per-token PK-prefix scan, as documented). Version column `ingested_at` → latest-wins, idempotent re-ingest. All match the Go row structs + INSERT column lists in `sink.go`.
- **Bloom skip-indexes correct + appropriate:** `idx_tx_hash`/`idx_tx_source` on transactions, `idx_op_source` on operations, `idx_contract_id` on contract_events — all `bloom_filter(0.01) GRANULARITY 1` on high-cardinality columns absent from the sort-key prefix, which is exactly when a bloom skip-index helps. GRANULARITY 1 (per-granule) is the standard, correct choice. Each is the column the matching reader filters on. Schema comments correctly flag the one-time `MATERIALIZE INDEX` needed for pre-existing history.
- **`op_index Int32`** on `ledger_entry_changes` holds the -1 tx/fee-level sentinel (every other table uses `UInt32` op_index — correct, since only entry-changes carry the -1 marker).
- **Partition key** `intDiv(ledger_seq, 1000000)` is uniform across all tables → 1M-ledger partitions, enabling partition-pruned ledger-range reads + bounded FINAL merge windows. Consistent.
- **i128 fidelity (ADR-0003):** `supply_flows.amount Int128` (not Int64) — no truncation. `total_coins`/`fee_pool` are `Int64` which is correct for XLM stroops (max ~5e16 < 9.2e18). `supplyFlowsDDL` const in `supply_flows.go` matches the canonical schema byte-for-byte.

**Other files:**
- **`gate.go` / `completeness.go` / `recognition.go`:** the substrate-continuity + hash-chain + watermark SQL is windowed (`substrateWindow = 5M`) to stay under the 12 GiB query-memory cap; `leadInFrame`/`lagInFrame` framing handles the last-row-no-spurious-gap edge; `toUInt64(ifNull(...,0))` normalises Nullable/width so the driver doesn't reject the scan; `watermark()` pure logic is unit-tested across 7 boundary cases. `from`/`to` are uint32 bound as `?`. The substrate window seam (`qlo--`) correctly re-checks the boundary pair. CORRECT.
- **FINAL discipline on streaming reads is principled:** counting consumers (`StreamContractEvents`, completeness reconcile, `supplySumQuery`, `DistinctTopicShapes` via GROUP BY) use FINAL or GROUP BY dedup; idempotent-downstream consumers (`StreamContractEventsFiltered` w/ `useFinal=false` from the live projector, `ReconcileEventStreamer` which dedups adjacent duplicates in O(1)) skip it deliberately, each with a docstring justifying the choice against the re-run-partition duplicate risk. Internally consistent.
- **`live_sink.go` G12-01/02 semantics:** PushLedger is non-blocking (drops on full channel, never back-pressures live ingest); `Sink.SetMaxBufferLedgers` bounds heap under sustained CH outage; `written` counts only post-Flush durability, `buffered` counts enqueue — the two-counter split correctly avoids over-claiming durability. `Flush` writes `stellar.ledgers` LAST as the per-ledger commit marker (load-bearing for `ContiguousWatermark`); buffer-full drops the whole extract atomically so no orphan ledgers row. Bounded-drop tested.
- **`extract.go` decode-at-ingest:** `claimAtomCount`/`realTradeCount` mirror the dispatcher + sdex census exactly (both-zero no-op crosses dropped; passive-offer dual-arm fallback) — table-tested per op variant. `eventRow` applies the same capture-eligibility gate as `dispatcher.captureEligible`. Supply-flow amounts decoded via `DecodeSupplyAmount` (i128 → `*big.Int`, ADR-0003) at ingest. Tx/event read errors counted (not silently swallowed) so undercounts are visible (G15-06).
- **`supply_flows.go` supply sum:** `sum(toInt256(...))` widens before summing so Σmint can exceed i128 without overflow, returned `toString` → parsed to `*big.Int` (`mustBig`, tested incl. i128-max). FINAL on the small per-contract key range. Total = mint − (burn + clawback). CORRECT.
- **`contract_call_op_reader.go` / `sdex_op_reader.go`:** `contractHex` from `strkey.Decode` (trusted) bound via `unhex(?)`; `position(base64Decode(body_xdr), unhex(?))` substring filter is a documented cheap SUPERSET (caller's `Decoder.Matches` is the exact predicate); successful-tx restriction via `tx_hash IN (… successful=1 …)` matches the census; `full_sorting_merge` join keeps the ops⋈results join memory-bounded; op body/result `SafeUnmarshalBase64` with per-row error context. CORRECT.

## Notes / non-findings deliberately not raised

- `DecodeSupplyAmount` (supply_decode.go) tries only `AsAmountFromI128` (signed) + the SEP-41 map variant, not `AsAmountFromU128`. This is CORRECT — SEP-41/CAP-67 mint/burn/clawback amounts are spec'd as `i128`, and the bare-i128 + map(amount) shapes are exactly the two the CLAUDE.md SEP-41 note describes. Not a bug.
- `OperationCount uint16` (schema `operation_count UInt16`) cast from `tx.OperationCount()` (int): Stellar caps ops/tx at 100, far under 65535. No realistic overflow.
- `RecentLedgers`/`LedgerBySeq` use FINAL on a `ledger_seq`-ordered table with no bloom dependency — FINAL is the right dedup here and does not defeat any skip-index (there is none on ledgers). Correct.
